# 离散数学助教Agent 修改意见

**日期**：2026/5/12  
**作者**：刘源昊

---

## 关于项目 (About)

该项目旨在构建可由教师自主扩展的智能助教 Agent 样例库，专门为老师教学课程提供服务，探索其在教学与科研中的应用模式，提升课堂互动性与教学支持能力，推动高校智能化教学改革的实践与推广。

具体来说，就是为我的老师上的离散数学这门课开发一个 Agent，通过搭建一个数据库，这个智能 Agent 要能够回答与课程相关的个性化问题，包括考试时间、课程要求、上课讲到的例题等等。但如果有学生问到一些知识点，这个 Agent 又可以调用通用的人工智能来解答。

---

## 1. 项目需求及现状

### 1.1 项目交付物
1. **（核心）** 一个基于 React 的完整 Agent 助教系统
2. 一个可与用户（学生与教师）交互的 WEB
3. 一个支持授课相关数据存取的长期，易维护的知识库

### 1.2 项目现状
*   **Agent**：
    1. 一个可支持查询授课信息，结合部分课内相关内容并调用 LLM 进行问题作答的 workflow。
    2. 一个已确定的数据库体系（Sqlite + ChromaDB）。
    3. 已打通的 RAG pipeline。
*   **WEB**：
    1. **后端**：实现了基本的，沟通 agent_core 与前端界面的简单逻辑。（FastApi Standard）
    2. **前端**：完成了主界面的 UI 设计，实现了一些简单的跳转逻辑。
*   **知识库**：目前的知识库依赖于非持久数据库，具体的存储方式，存储形式等还有待定义。

## 2. 目前的不足

> 注：由于之前的工作一直围绕 agent 展开，本节只讨论 agent 存在的不足。重要性由高到低排列。

### 2.1 整体框架设计缺陷
项目目前并未兑现“Agent”这一理念。Agent 是以 LLM 为大脑进行决策的智能体，具有完整的 skill-tool 体系，由大模型自主决策并回答用户问题。但目前的“Agent”更类似于 if-else flow。

缺少 LLM-planner 的前提下，目前的架构更类似于 prompt 注入 + LLM 调用，缺少了 LLM reason 的环节，相当于我们替 LLM 进行了决策。

在离散数学助教的场景下，缺乏 LLM reasoning 最致命的缺陷就是“答非所问”。

**典型例子：**
*   **学生问**：空集里什么元素都没有，它为什么是任何元素的子集？
*   **当前逻辑**：意图识别为教学内容类 -> 查询为第一章内容 -> 向量检索（假设可以检索到） -> 将第一章第5页 PPT 内容送给 LLM -> LLM 收到较多上下文，有概率回答的内容完全没有结合或者强行结合不相干 PPT 内容进行作答。
*   **结果**：用户收到了来自 agent 的“胡言乱语”，或者 agent 降级为普通 LLM。

有了 LLM-reasoning，它可以及时发现 PPT 与内容不相干，并要求其返工或重新思考，从而避免这种情况。

### 2.2 定义不明确，解决方案不到位
项目暂未形成一个“提出问题”——>“设计工具”——>“解决问题”的理念，而是针对需求定制解决路径，各路径之间高度解耦，可复用性差且混乱。

例如，目前的硬过滤系统针对“课程信息”查 sqlite 返回固定内容。如果后续要拓展项目（如增加其他老师），目前的硬过滤系统将失效，且方案过于机械。

**建议方案**：把查库打包成工具，依托 LLM 判断是否调用这个工具。

### 2.3 RAG 能力较差
目前仅使用了稠密检索（余弦相似度计算）。
*   **效率问题**：计算效率随文档数量线性下降，可能导致内存爆炸。
*   **准确性问题**：对于词频小、专业性高的词汇（如：反自反性，最小点割集等），稠密检索排序不准。

**建议方案**：增加粗排（用词袋模型先筛），再用精排精确匹配。

### 2.4 没有完整的日志及错误追踪机制
项目几乎没有调试信息和日志，这对查 bug 和找漏洞是致命的。

### 2.5 数据存储形式，方式，内容不明
没有一个 stable，持久的存储库。没有严格定义数据存储格式（例如教学方案直接写进 sqlite，PPT 切片没有元数据等）。

## 3. 修改意见及 Todo

### 整体修改意见
建议先明确职责与边界，先将整体项目划分为 **agent_core + 后端 + 前端** 三部分。
*   **agent_core**：用 submodule 单独隔离出来，只预留 interface 和后端沟通。
*   **后端**：完成与前端交互、数据持久化存储、agent 互通。
*   **前端**：根据需求设计界面，确保安全。
